Hard and soft isolation

So far, some kind of "mount" just isolates the DOM totally - you can't use selectors to get from the outside to the inside even if you wanted to because selectors don't understand how to cross anything but parent/child links. Is that a problem?

The commonly cited use-case is that the outside page wants to providing 'theming' advice for the stuff inside our special isolated world. In other words, if I create a 'calendar' and it's got a nice blue theme and then you want to use it on your site which for some reason your designer convinced you should be orange - that just might cause you to think "yeah, but I'll need to customize it". This example seems so basic and so patently obvious, and we're so caught up in our frustration with native elements not being stylable that I think a lot of us immediately jump to to "yes of course you want to be able to cross the boundary and make it blue - at least it would do something safe by default and that's a pretty big deal. At a minimum, this is where I've always been on the subject - until recently, and now I'm not so sure.

Let me explain why. In a nutshell, it's because leads to inevitable follow up questions that I can't answer well - a whole series of potential footguns when I really think about it. It clearly is better than what we have now - there's no doubt about that, and that makes it tempting to say "good enough" -- but our current world doesn't have so many custom elements or isolation and thus degree of complexity we are able to deal with is somewhat limited by this fact. Adding some kind of isolation and custom elements means we'll do it more and more and build increasingly complex things. Now imagine that you create something like, panel with a titlebar which contains various buttons. Of course you should be able to make the titlebar blue, so it's really tempting to say, no problem, if you consciously cross the boundary you can say something like x-panel/mount/.titlebar { background-color: blue; }. But this case is so trivial that it's easy to look past the issues with that. What happens when my panel hosts another element which uses .titlebar for something else and that thing -shouldn't- be blue? I guess you'll want to refine your selector say 'only ones that are a child of the mount' likewise you'll want to set your own 'close' icon and you have a similar problem, so you'll want to say something like x-panel/mount/>.titlebar>.closeIcon. As you start getting into real cases you see that you're getting pretty coupled with my implementation. If I decide I need to add an element in between .titlebar and .closeIcon, I'm going to break your styles.

Ah, but if you could create pseudo-elements this would be solved, right? That's always been my answer as well, but again, I'm not so sure it would. Differentiating them requires some context and what about custom elements inside of other custom elements? When you use it, are you actually grabbing a element? Does that mean you have full powers to everything inside it? Getting this right actually seems pretty confusing. I'm not saying we _can't_ figure out the right way, I'm saying that I can't currently reason about it and I don't know

It seems to me that when I stop and think about it - if you can actually touch this from the outside with CSS, inaddition to doing something innocent, you could do something disasterous -- you could actually break my 'panel-ness' for example. Consider if we had only 1 base element and we had to build everything out of that. If you were to build the select element you'd create the options dropdown full of block things and have it hide by default, this would have carefully constructed meaning to UAs by way of both CSS and ARIA. These things together define it's select-ness... Without them, they cease to be a select. By simply twiddling a CSS property incorrectly you could break an important aspect and make it no longer 'it'. Background-color, of course, is not generally one of those things that could fundamentally break something - but lots in CSS is that way: The display property is kind of fundamental, for example.

So, it's clear you have to be able to provide theming advice from the outside to the inside, but it's unclear how that should work or should be standardized. At the heart of this, if you step back, it seems that a component should choose what things about itself are stylable and what are not. For simiplicy sake then, let's just say "provide very targeted theme advice for components via properties set in script"
